# Judge Criteria (1 条)

## 拆书分析内容质量标准 [basic]

### 评分标准（matched: true/false）

**【重要】本检查项的范围约束**：
- ✅ 只检查：拆书分析文档（novel_analysis.json）的**内容质量**是否符合业务规则Step 2的要求
- ✅ 检查3个关键维度：名场面标注、人物分析主次区分、情感引擎判定
- ❌ 不检查：文件格式/Schema合规性（那是格式检查"拆书分析格式"的范围）
- ❌ 不检查：文件是否存在（那是"拆书分析文件存在"的范围）
- ❌ 不检查：拆书分析与策划/剧本的一致性（那是跨文件一致性检查的范围）

**本检查项与其他检查的关系**：
- "拆书分析格式"(F1) → 只管 JSON 格式和 Schema 合规
- "拆书分析文件存在"(B1) → 只管文件存在
- "拆书分析内容质量"(本项) → 管内容是否有价值、是否满足业务规则要求

**不合格（matched: false）：内容质量不达标**

**典型问题1：名场面缺乏原文位置标注**
- 业务规则要求：名场面清单中"每个标注原文章节位置"
- 不合格示例：
  ```json
  "key_scenes": [
    {"name": "初次相遇", "description": "男女主在图书馆邂逅"},
    {"name": "表白场景", "description": "男主在雨中告白"}
  ]
  ```
- 问题：没有标注这些场面在原著中的位置（哪一章？哪个段落？）
- 合格示例：
  ```json
  "key_scenes": [
    {"name": "初次相遇", "description": "男女主在图书馆邂逅", "source_location": "第3章'偶遇'开头段落"},
    {"name": "表白场景", "description": "男主在雨中告白", "source_location": "第12章末尾至第13章开头"}
  ]
  ```
- 评估标准：业务规则要求"每个标注原文章节位置"——检查名场面是否都有某种形式的位置标注（章节号、段落描述、原文关键句引用等任一形式均可）

**典型问题2：人物分析未区分主次**
- 业务规则要求："人物分析必须区分主要/次要，次要人物不能过多"
- 不合格示例：
  ```json
  "characters": [
    {"name": "林沐"}, {"name": "顾言"}, {"name": "张浩"},
    {"name": "李薇"}, {"name": "王叔"}, {"name": "赵秘书"},
    {"name": "陈老板"}, {"name": "刘助理"}
  ]
  ```
- 问题：8个角色全部平铺，没有区分谁是主角、谁是配角、谁只需一笔带过
- 合格示例：
  ```json
  "character_analysis": {
    "main_characters": [
      {"name": "林沐", "role": "女主", "traits": "...", "arc": "..."},
      {"name": "顾言", "role": "男主", "traits": "...", "arc": "..."}
    ],
    "supporting_characters": [
      {"name": "张浩", "role": "反派", "function": "推动女主觉醒"}
    ],
    "minor_mentions": ["王叔", "赵秘书"]
  }
  ```
- 评估标准：存在某种形式的主次区分（main/supporting 分类、role_type 标注、或独立的主要/次要分组均可）

**典型问题3：情感引擎判定缺失或不准确**
- 业务规则要求："情感引擎判定要准确，它决定了后续的节奏风格"
- 不合格情况1：完全没有情感引擎/emotional_engine/主导情绪相关字段
- 不合格情况2：情感引擎判定与原著实际内容明显不符
  - 示例：原著是悬疑推理题材，情感引擎却标注为"甜宠"
  - 示例：原著是校园热血，情感引擎标注为"虐恋"
- 合格标准：有情感引擎相关字段，且判定结果与原著的主要情感基调基本一致
- 注意：不要求用字段名完全匹配，任何表达主导情绪/情感类型的字段都算（如 genre, tone, emotional_core, 情感引擎, 题材类型 等）

**合格（matched: true）：三个维度基本满足**
- 名场面清单中的名场面有原文位置标注（业务规则要求"每个标注原文章节位置"）
- 人物分析有明确的主次区分机制
- 情感引擎/主导情绪有明确判定且与原著基调匹配
- 不要求字段名与示例完全一致，关注的是**信息是否存在且合理**

**验证要点**：
1. **读取拆书分析** (novel_analysis.json)：检查整体结构和内容
2. **名场面检查**：
   - 是否有名场面/key_scenes类似的列表？
   - 每个名场面是否有位置信息（章节、段落、原文引用等）？
   - 业务规则要求每个都有，但允许个别遗漏——大部分有即可
3. **人物分析检查**：
   - 是否有角色/character分析？
   - 是否区分了主要和次要角色？
   - 次要角色数量是否合理（业务规则要求"次要人物不能过多"）？
4. **情感引擎检查**：
   - 是否有情感引擎/主导情绪/题材类型相关字段？
   - 判定结果与原著（source_novel.md）的实际内容是否匹配？
5. **对比原著** (source_novel.md)：
   - 名场面标注的位置是否与原文对应？
   - 情感引擎判定是否准确？

**判断流程**：

**第一步（必须执行）**：请先通读参考原著，形成对原著的基本认知：(1)故事的主要情感基调/题材类型（如甜宠、虐恋、悬疑、热血等）(2)原著中有哪些名场面/高潮段落，它们大致在哪些章节 (3)主要角色有哪些，谁是主角谁是配角。将提取结果记录在返回JSON的 `novel_baseline` 字段中。
> 注意：只有先读完原著，才能判断拆书分析中的名场面位置标注是否准确、情感引擎判定是否正确。

**第二步**：基于第一步的原著认知，逐一检查拆书分析的三个维度。
1. 检查名场面位置标注 → 是否与原著实际位置对应？合格/不合格
2. 检查人物分析主次区分 → 合格/不合格
3. 检查情感引擎判定 → 与原著实际基调是否匹配？合格/不合格
4. 三个维度中：
   - 3项全合格 → matched=true
   - 2项合格 → matched=true（有瑕疵但整体可用）
   - ≤1项合格 → matched=false
5. 如果某个维度完全缺失（连相关字段都没有），直接判该维度不合格

请回复JSON格式：
{"matched": true/false, "novel_baseline": {"genre": "题材类型", "key_scenes_in_novel": ["场景1@章节位置", "场景2@章节位置"], "main_characters": ["主角1", "主角2"]}, "scene_location_check": {"total_scenes": 5, "scenes_with_location": 4, "ratio": "80%", "passed": true}, "character_hierarchy_check": {"has_hierarchy": true, "main_count": 3, "supporting_count": 2, "method": "main/supporting分类", "passed": true}, "emotional_engine_check": {"has_engine": true, "engine_value": "复仇/逆袭", "matches_novel": true, "passed": true}, "dimensions_passed": 3, "reason": "说明三个维度的检查结果"}
